## タスク詳細

## P0-1: `alert_email.py` 銘柄コード変換の堅牢化
### 変更対象
- `alert_email.py` の `_row_code_str` とその呼び出し箇所

### 実装指示
- 現在の `int(row["銘柄コード"])` 直変換を廃止し、以下を満たす安全変換に置換する。
  - `None` / `NaN` / 空文字は `None` を返す。
  - 文字列は `strip()` して数値判定する。
  - 変換不能値は例外を投げず `None` を返す。
- 呼び出し側は `None` を受け取ったら当該行をスキップし、処理継続する。

### 受け入れ条件
- 不正コードを含む ranking CSV を与えても `alert_email.py` が落ちない。
- スキップ件数がログ出力され、処理全体は完了する。

---

## P0-2: 異常入力耐性テストの追加
### 変更対象
- `tests/test_alert_email.py`

### 実装指示
以下のテストケースを追加する。
1. `銘柄コード` が `NaN` の行を含む。
2. `銘柄コード` が空文字の行を含む。
3. `銘柄コード` が数値以外（例: `ABCD`）の行を含む。
4. `ネット(%)` 欠損時のフォールバック挙動。
5. `下落確率(%)` が文字列のときも候補抽出が継続する。

### 受け入れ条件
- `python3 tests/test_alert_email.py` が成功する。
- 上記ケースで「例外が発生しない」ことを明示的に検証している。

## P1-1: README と実装の仕様整合（決算ルール）
### 変更対象
- `README.md`
- `alert_email.py`（定数名・コメント）

### 実装指示
- ルールを以下2つに分離して明記する。
  - **表示ルール**: 決算接近時にバッジ表示する条件
  - **除外ルール**: 新規候補から除外する条件
- 定数名は責務別に分ける（例: `EARNINGS_WARN_DAYS`, `EARNINGS_EXCLUDE_DAYS`）。

### 受け入れ条件
- READMEを読んだ仕様とコード挙動が一致する。
- PRレビュー時に「14日? 7日?」の解釈ブレが発生しない。

## P2-1: 段階的リファクタ（任意だが推奨）
### 第1段階: 責務分割
- `services/ranking_loader.py`
- `services/candidate_selector.py`
- `services/mail_renderer.py`
- `services/mail_sender.py`

### 第2段階: 設定集約
- `config.py` を作成し `Settings` dataclass でしきい値と環境変数を一元管理。

### 第3段階: 型導入
- `HoldingResult`, `RankingRow`, `Candidate` を dataclass 化。

### 第4段階: I/Oアダプタとログ
- 外部I/Oを adapter 化し、`logging`（JSON形式推奨）へ移行。

### 受け入れ条件
- 既存機能を壊さず、ユニットテストが維持・増加する。
- 主要ロジックが I/O なしでテスト可能になっている。
